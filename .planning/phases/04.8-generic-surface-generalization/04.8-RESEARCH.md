# Phase 04.8: Generic Surface Generalization - Research

## User Constraints

### Locked Constraints

- Do not re-enter an endless benchmark-fix loop.
- Do not solve the `20`-run by adding infinite new topic families.
- Keep room for later rediscovery; do not pretend this phase locks the final architecture forever.
- Prefer a small number of reusable patterns over case-by-case benchmark rescue.

## Phase 04.8 Must Deliver

- A bounded generalization of the non-promptless answer surfaces where the `20`-run still falls back to prompt-style reasoning.
- A clear statement of what surface/routing behavior becomes generic after this phase.
- Local proof and failed-cluster proof strong enough to justify either:
  - one more broader paid rerun, or
  - a clean handoff to MemPalace adoption planning if the remaining limit is substrate-level.

## Current Code Reality

- Phase 04.7 made promptless authoritative answers externally comparable and canary-clean.
- The paid `10/10` closure was real for the sampled baseline.
- The wider paid `20`-run still failed on `8` cases with a clustered pattern:
  - aggregate totals / thresholds
  - event ordering / temporal compare
  - abstention surfaces
- The current weakness is no longer obvious kernel-correctness collapse and no longer promptless-authoritative compatibility. It is generic non-promptless surface generalization.

## Recommended Implementation Shape

- Keep the work at the surface/routing layer first.
- Reuse existing structured state and fact substrates wherever possible.
- Add only the smallest new generic contracts needed to stop falling back to weak prompt reasoning.
- Treat the failed `20`-run questions as a proof cluster, not as individual target APIs.

## Best 3-Plan Split

### Plan 04.8-01

Define the generalized structured surface contracts and routing boundaries for:

- aggregate totals / thresholds
- event ordering / temporal compare
- canonical abstention

### Plan 04.8-02

Implement the generalized surfaces and routing in Core2 without topic-specific family explosion.

### Plan 04.8-03

Prove the generalized surfaces against the previously failed `20`-run cluster plus local handmade/golden-path anchors, and define the rerun stop rule.

## Testing Guidance

- Reuse the existing Core2 test stack; do not create a second benchmark framework.
- Add only the smallest new coverage needed to prove the three generic surface classes.
- Require that the failed `20`-run cluster becomes explainable by explicit surface classes, not by narrative reasoning.
- Add a tiny negative-proof mini-set for lookalike cases that must still fail closed, so surface generalization cannot silently turn into soft matching.
- Paid rerun trigger rule for this phase:
  - local handmade acceptance green
  - targeted generalized-surface tests green
  - the previously failed cluster is locally resolved or explicitly fail-closed
  - only then a broader paid rerun

## Risks And Failure Modes

- The phase could silently degrade into topic-family enumeration. That would be failure, even if some benchmark items turn green.
- The phase could overreach and start changing retrieval substrate concerns that properly belong to the later MemPalace work.
- The phase could hide weak answers behind broad surface labels unless local proofs stay strict.
- The phase could become another comparator project. That is explicitly out of scope.
- The phase could introduce routing churn if precedence between existing promptless surfaces, new generalized surfaces, and fallback is not fixed explicitly.

## Open Questions

- Which currently structured facts are already sufficient to support the generalized aggregate/ordering/abstention surfaces, and where is bounded composition enough?
- Does event ordering need a dedicated normalized event packet, or can it be built from current event facts plus stricter routing?
- After the failed cluster is green locally, should the next paid rerun be another `20`, or a different broader confirmation set?
