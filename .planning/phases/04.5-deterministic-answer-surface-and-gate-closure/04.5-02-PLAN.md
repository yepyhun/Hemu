# Plan 04.5-02: Prefer The Answer Surface In Hermes Handoff

## Objective

Make the real Hermes handoff and direct-answer path prefer the new deterministic Core2 answer surface so covered durable-memory cases stop depending on loose prompt formatting and prompt drift.

## Must Haves

- The handoff path stays thin and provider-first.
- Structured-only guards remain active: direct behavior must come from proven structured surfaces, not from new ad hoc heuristics.
- The path remains aligned with the existing Core2 provider seam and does not become benchmark-only code.

## Tasks

- Wire the provider/runtime output so the final answer path can consume the deterministic answer surface before broader answer synthesis.
- Ensure `core2_authoritative`, provider glue, and any thin native handoff points agree on the same surface contract.
- Preserve explicit packet metadata so later debugging can tell whether the final answer came from:
  - deterministic answer surface
  - structured fact-first path with explanatory support
  - broader fallback path
- Keep broader fallback behavior available when the answer surface is absent or insufficient, but do not let covered families skip directly to broad prompt-shaped reasoning by default.
- Keep the changes concentrated in Core2 modules and thin handoff wiring; avoid spreading new logic through unrelated runtime files.

## Verification

- Focused tests prove that covered families prefer the answer-surface path.
- Structured-only guards still prevent unsupported direct answers.
- Existing Hermes/Core2 runtime proofs remain green after the handoff wiring change.
