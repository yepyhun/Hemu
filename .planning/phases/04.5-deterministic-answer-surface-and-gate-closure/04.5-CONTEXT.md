# Phase 04.5: Deterministic Answer Surface And Gate Closure - Context

## Phase Boundary

Phase 04.5 exists because Phases 04.2, 04.3, and 04.4 materially improved the Core2 substrate, but the real Hermes-path gate still shows `prompt_miss` as the dominant failure pattern. The remaining problem is no longer missing durable-memory families; it is the handoff boundary between Core2 and the final answer path.

This phase is therefore about making Core2 emit a narrow, provider-owned answer surface for covered durable-memory families and making the remaining live-gate failures mechanically classifiable. It is not another benchmark-only phase and it is not a license to add more loose query-time heuristics.

## Implementation Decisions

### Locked Decisions

- This phase is handoff-structural, not benchmark-cosmetic.
- `plan7vegrehajt.md` remains the primary product contract. `plan6.md` remains the anti-loop and anti-collapse guardrail.
- The solution must stay inside the existing Core2 architecture and provider seam. Do not build a second runtime or a sidecar memory engine.
- The answer surface must be provider-owned and inspectable. It cannot be a hidden prompt blob assembled ad hoc in the benchmark runner.
- The answer surface must have an explicit small-size ceiling so it cannot quietly grow back into a prompt-shaped recall blob.
- The deterministic surface must reuse the fact-first substrate from Phases 04.2-04.4. It must not reintroduce raw-blob-first reasoning as the primary path.
- Remaining gate failures must be classified mechanically into at least:
  - kernel correctness
  - handoff/format
  - latency
- The classifier must fail closed: if a case cannot be assigned confidently, it must surface as `unknown` or equivalent rather than being forced into a cleaner-looking bucket.
- This phase must not recreate the old kernel collapse by adding semantic-contract explosion, giant graph orchestration, or benchmark-specific rule forests.
- Phase 04.1 should resume only after 04.5 proves that the new handoff path is active and that the remaining external-gate blockers are explicit.

### At The Agent's Discretion

- The exact shape of the answer-surface object, as long as it stays narrow, typed, and inspectable.
- Whether the surface lives mainly in `core2_types`, `core2_answer`, `core2_runtime`, or the provider layer, as long as responsibilities stay clean.
- The exact names of gate-matrix categories and status fields, as long as they are stable and machine-readable.

## Specific Ideas

- Introduce one narrow answer-surface packet for covered durable-memory families:
  - family
  - answer mode
  - deterministic first-pass text or compact structured answer fields
  - provenance/support metadata
  - explicit fallback hint when the surface is insufficient
- Let the provider/runtime emit this surface before the final answer path rather than forcing the final model to rediscover the same shape from a recall blob.
- Keep the structured-only guard: direct answer behavior must only activate when packet metadata proves the answer came from the fact-first substrate or an equivalent structured Core2 artifact.
- Make the gate status artifact classify failures by blocker family so later work is not guided by vague “still not good” judgments.
- Keep one tiny frozen set of real failing questions as a canary replay set before the paid rerun, so handoff regressions are caught cheaply and repeatedly.

## Deferred Ideas

- Broad multilingual universal answer-surface rendering
- General-purpose summary generation outside the covered durable-memory families
- Giant graph-first answer planning
- Any new benchmark-only branch that bypasses the real Hermes/Core2 handoff
