# Phase 04.10: Bounded Hybrid Substrate - Research

## User Constraints

### Locked Constraints

- The user wants the `04.9` recommended branch, not a full MemPalace switch.
- The hybrid must stay implementable and not collapse under ownership complexity.
- The phase should improve the lower layers where MemPalace looks strongest, not reopen the already-built deterministic Core2 hard core.
- Retrieval gains alone are not enough; end-to-end memory correctness remains the acceptance standard.
- Governance-heavy memory infrastructure is explicitly out of scope for this phase.
- Behavioral/session-mining or profile-synthesis systems, if ever useful, belong to a later sidecar lane rather than this substrate phase.

## What 04.9 Actually Authorized

From Phase 04.9:

- strongest likely seam:
  - lossless raw substrate
  - scoped semantic retrieval
- Core2-owned layers to preserve:
  - truth/state/provenance
  - answer surface
  - abstention
- optional later layers, not for this phase:
  - graph/navigation
  - temporal KG as anything more than secondary support
  - AAAK-heavy rollout

## Recommended Implementation Shape

Make the smallest meaningful hybrid move:

1. define the substrate adapter seam and ownership model
2. introduce MemPalace-inspired raw/verbatim storage plus scoped retrieval
3. prove that Core2 truth/state and answer behavior do not regress

This phase should not try to solve every MemPalace idea at once.

## Best 3-Plan Split

### Plan 04.10-01

Define the bounded adapter seam and ownership rules.

### Plan 04.10-02

Implement the hybrid raw substrate and scoped retrieval path.

### Plan 04.10-03

Prove non-regression against Core2 truth/answer behavior and define cutover rules.

## Testing Guidance

- Local proof must separate:
  - substrate behavior
  - retrieval behavior
  - truth/answer regression
- Phase proof is not “higher benchmark score at any cost”.
- The strongest verification is:
  - new substrate path exists
  - scoped retrieval works
  - existing deterministic truth/answer tests remain green
- Do not treat retrieval-only improvements as a substitute for end-to-end memory validation.
- Add one or two paired retrieval traces that show the same query through the old path and the new hybrid path.
- Define rollback behavior early so introducing the hybrid path does not turn debugging into an all-or-nothing migration.

## Risks And Failure Modes

- adapter sprawl that leaks into truth logic
- dual-read/dual-write confusion
- hidden benchmark-driven changes in answer behavior
- substrate work that silently changes abstention or answer-surface semantics
- overcommitting to graph/KG/compression too early
- letting MemPalace naming or taxonomy leak upward into Core2 contracts
- no clean rollback if the hybrid path behaves worse in practice
- chasing unrelated governance or profile-mining ideas mid-phase and losing the bounded seam

## Open Questions

- Should the first hybrid step write to a new substrate only, or read from it only, or do both in a bounded path?
- Which current Core2 retrieval calls are the cleanest insertion points for scoped substrate search?
- Should a broader paid rerun happen before or after a first bounded substrate implementation slice?
