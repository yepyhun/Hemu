# Plan 04.6-01: Define The Deterministic Core Boundary

## Objective

Define exactly what Core2 claims as hard deterministic kernel behavior, and what it explicitly leaves outside that boundary.

## Must Haves

- A short explicit list of kernel axioms/invariants
- A clean boundary between deterministic core and heuristic layer
- Mapped code anchors for each hard invariant
- No fake promise that raw-language understanding is fully deterministic

## Tasks

- Write a short `CORE2-AXIOMS.md` reference artifact that names the hard deterministic kernel guarantees.
- Write a short `CORE2-GLOSSARY.md` so the phase vocabulary stays stable across planning, implementation, and external review.
- Define the minimum invariant set for v1:
  - fact identity/type/key
  - provenance
  - current/previous/superseded state
  - conflict state and escalation
  - deterministic supersession rules
  - covered fact-first recall
  - fail-closed answer surface
  - abstention on insufficient structured support
- Define the explicit heuristic/outside layer:
  - candidate extraction
  - broad semantic matching
  - relation/graph proposal
  - summarization
  - fuzzy ranking for weak matches
  - linguistic phrasing
  - benchmark judge
- Map each hard invariant to concrete Core2 code anchors and test anchors.
- Update project requirements/docs so the deterministic boundary becomes a first-class project contract rather than an implied understanding.

## Verification

- The axioms are short enough to be auditable by a human reviewer.
- Every hard invariant maps to real code paths or tests.
- The document makes it clear what Core2 does not claim to solve deterministically.
- The glossary is small and eliminates the current recurring terminology ambiguities.
