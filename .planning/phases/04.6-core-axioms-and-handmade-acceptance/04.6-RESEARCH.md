- current/previous/superseded semantics

The current problem is no longer “Core2 has no architecture.” The problem is that the remaining finish-line decisions are still too implicit:

<external_guidance>

## External Pattern Guidance

The recent external advice points in one narrow, useful direction:

- define the axioms or logic under which “correct” is judged
- constrain the problem space before deeper linking/reasoning explodes in dimensionality
- keep a small mechanically verifiable core instead of pretending everything is fully formal
- use a handmade benchmark that satisfies the human operator before relying on broader external evals

This supports a small deterministic core plus bounded acceptance artifacts. It does not justify a total architecture restart.

</external_guidance>

<recommended_structure>

## Recommended Implementation Shape

Make one bounded clarification layer:

1. Define the hard deterministic core boundary.
2. Define a small handmade acceptance benchmark that exercises it.
3. Define a fail-closed classification matrix for the live gate.
4. Resume 04.1 only after those three are explicit.

The likely deterministic core is:

- stable fact identity/type/key
- provenance/source lineage
- temporal state (`current`, `previous`, `superseded`)
- conflict state and escalation
- deterministic supersession rules
- covered fact-first retrieval behavior
- fail-closed answer surface behavior
- abstention when structured support is insufficient

The likely heuristic/outside layer is:

- raw text to fact-candidate extraction
- broad semantic matching
- relation/graph proposal
- summarization
- fuzzy ranking for weak matches
- final style phrasing
- external benchmark judge behavior

The important boundary is after candidate extraction: Core2 should not promise to fully formalize raw natural-language understanding, but it should promise deterministic handling once structured candidate objects exist.

</recommended_structure>

<testing_guidance>

## Testing Guidance

- The handmade benchmark should be small, explicit, and human-readable.
- Use existing Core2 proof anchors where possible; do not spin up a second sprawling test stack.
- Best anchors:
  - `tests/agent/test_core2_fact_digestion.py`
  - `tests/agent/test_core2_retrieval_routing.py`
  - `tests/agent/test_core2_authoritative.py`
  - `tests/agent/test_core2_longmemeval_benchmark.py`
  - `.planning/phases/04.1-longmemeval-gate-and-performance-fixes/04.1-GATE-STATUS.json`
- The handmade acceptance set should include:
  - exact expected answer or abstention
  - exact structured support expected to exist
  - explicit pass/fail meaning
- The gate matrix should require `unknown` when evidence is insufficient to classify a miss cleanly.
- Add a paid-gate rerun trigger rule: do not spend the paid gate unless the handmade set is green and the current change is mapped to an explicit blocker class.
- Add one or two golden-path traces as debugging/reference artifacts so future regressions can be compared to a known-good ingest -> fact -> recall -> answer path.
- Add a tiny glossary so the phase stops losing time to terminology mismatches (`family` vs deeper knowledge-structure terms, `judge artifact`, `handoff miss`, etc.).

</testing_guidance>

<risks>

## Risks And Failure Modes

- If the deterministic core is drawn too broadly, the phase will become a fake formal-method rewrite.
- If it is drawn too narrowly, it will not actually constrain the final 04.1 decisions.
- If the handmade benchmark is vague or LLM-judged, it will not solve the current steering problem.
- If the gate matrix lets ambiguous misses be relabeled too optimistically, the project will drift back into prose-only “almost done” judgments.

</risks>

<open_questions>

## Open Questions

- Should the handmade acceptance set live only as markdown, or also as a tiny machine-readable fixture for repeatable local runs?
- Which covered families must be in the handmade set for v1 confidence, and which can be deferred?
- How much of relation/graph behavior should be named explicitly in the axioms versus left as an implementation detail?

These are implementation choices, not blockers. The phase should aim for the smallest explicit contract that still makes 04.1 trustworthy.

</open_questions>
