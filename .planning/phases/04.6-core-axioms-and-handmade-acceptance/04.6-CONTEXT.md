# Phase 04.6: Core Axioms And Handmade Acceptance - Context

**Gathered:** 2026-04-07
**Status:** Ready for planning

<domain>

## Phase Boundary

Phase 04.6 exists because the project now has a real Core2 substrate, fact-first recall, and deterministic answer surfaces, but the finish line is still too dependent on the paid live gate and its external judgment path.

This phase must define:

- the smallest hard deterministic core that Core2 claims as real kernel behavior
- a small handmade acceptance set that proves local architectural progress before LongMemEval
- a fail-closed gate classification matrix for the remaining live-path misses

This phase is not:

- a full formal-method rewrite
- a Lean/Haskell port of the kernel
- another memory-family expansion phase
- the final paid LongMemEval rerun itself

</domain>

<decisions>

## Implementation Decisions

### Locked Decisions

- This phase is architectural clarification and acceptance hardening, not feature expansion.
- `plan7vegrehajt.md` remains the primary product contract. `plan6.md` remains the anti-loop and anti-collapse guardrail.
- The hard deterministic core must stay small. Do not pretend the entire memory intelligence stack can be made fully formal in v1.
- The deterministic boundary must be defined around structured kernel objects and state transitions, not around universal raw-language understanding.
- The handmade acceptance set must be written without LLM judging as its primary arbiter. It should act as a local compass for “are we moving in the right direction?”
- The gate matrix must be fail-closed. Any miss that cannot be cleanly classified must stay `unknown`, not be force-fit into a nicer label.
- Reuse the existing `agent/core2_*` modules, tests, gate artifacts, and provider seam. Do not build a parallel review or benchmark framework beside Core2.

### the agent's Discretion

- Exact wording of the core axioms and invariant statements
- Whether the handmade benchmark lives as a markdown spec only, a machine-readable fixture, or both
- Whether the gate matrix should be documented only, or also reflected in a tiny machine-readable schema

</decisions>

<specifics>

## Specific Ideas

- Create a short `CORE2-AXIOMS.md` that names exactly what the kernel guarantees deterministically.
- Create a short `CORE2-HANDMADE-BENCH.md` that captures 8-12 real memory cases with explicit expected outcomes.
- Create a short `CORE2-GATE-MATRIX.md` that constrains every remaining miss to:
  - kernel correctness
  - handoff/format
  - judge artifact
  - unknown
- Make the handmade benchmark the local “directional proof” before spending the paid gate again.
- Keep the kernel/heuristic boundary explicit:
  - candidate extraction can stay heuristic
  - structured state, supersession, conflict, abstention, and covered fact-first behavior must be deterministic
- Add a tiny `CORE2-GLOSSARY.md` so terms like `family`, `knowledge structure`, `candidate`, `answer surface`, and `handoff miss` do not drift across sessions or external reviews.
- Add one or two tiny golden-path traces that show the intended happy path from ingest to answer surface for a canonical durable-memory case.

</specifics>
