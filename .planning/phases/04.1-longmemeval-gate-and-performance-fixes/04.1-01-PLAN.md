# Plan 04.1-01: Diagnose Live Failure Patterns And Telemetry

## Objective

Make the LongMemEval live path diagnosable enough that another failed run immediately tells us whether the problem is in retrieval, answer synthesis, grounding handoff, judging, or wrapper overhead.

## Must Haves

- Replayable evidence from the failed live path
- Per-item timing split for seed, retrieval/tool, answer, judge, and total
- Per-item token reporting for prompt, completion, and estimated baseline replay
- Stable failure categories rather than opaque pass/fail only
- Smoke-gate and early-abort behavior that can stop obviously bad runs fast

## Tasks

- Inspect `agent/core2_longmemeval_benchmark.py` and identify the exact boundaries where timing spans should begin and end.
- Capture or reconstruct a small set of live-run failure cases that can be replayed locally through the Hermes path.
- Add explicit timing capture around:
  - dataset seed/setup
  - Core2 retrieval/tool usage
  - `run_conversation` answer generation
  - judge call
  - whole item wall time
- Normalize the result schema into a concise per-item record and aggregate summary.
- Strengthen failure taxonomy so it distinguishes at least:
  - abstention
  - prompt miss
  - grounding handoff miss
  - retrieval or reasoning miss
  - answer present in prompt but wrong final output
  - timeout or latency abort
- Add targeted tests proving the replayed failure cases are inspectable and classifiable.
- Add a smoke-gate mode with configurable thresholds and report why a run stopped.
- Add or adapt tests around telemetry schema, failure classification, and early-stop behavior.

## Verification

- `python3 -m py_compile agent/core2_longmemeval_benchmark.py scripts/run_core2_longmemeval_subset.py`
- `.venv/bin/python -m pytest tests/agent -k 'core2 and (proof_harness or hardening or longmemeval or benchmark)' -n 0`
- Run a tiny local benchmark smoke case and inspect that the JSON report includes the expected timing and failure fields.
