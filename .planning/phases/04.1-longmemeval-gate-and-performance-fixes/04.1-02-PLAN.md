# Plan 04.1-02: Fix Kernel Correctness Gaps And Add Safe Latency Controls

## Objective

Fix the kernel behaviors exposed by the live gate and only then bring the benchmark path closer to the user's expected latency envelope without cheating the real Hermes-integrated route.

## Must Haves

- The run must still go through Hermes with `memory.provider=core2`
- Live failure cases must become regression tests before the larger paid rerun
- Avoidable benchmark overhead must be removable or measurable
- Any fast-path must be explicit and benchmark-scoped, not a hidden behavior change

## Tasks

- Trace the current answer-generation path from the benchmark harness into `run_agent.py` and identify correctness and latency risks visible on the real path.
- Implement fixes for the concrete failure cases surfaced by Plan 04.1-01, prioritizing:
  - answer synthesis after successful retrieval
  - grounding/provenance propagation into the final answer
  - route-specific behavior that only appears through Hermes `run_conversation`
- Prefer thin benchmark-specific controls over broad runtime redesign, for example:
  - benchmark-mode config patching
  - narrower answer style
  - explicit non-essential feature suppression during benchmark runs
  - optional non-streaming or lower-overhead answer mode if it preserves correctness
- Keep Core2 retrieval semantics unchanged unless the telemetry or replayed failure cases prove retrieval is materially involved in the correctness problem.
- Ensure the runner can execute a fast smoke gate before any larger paid sample.
- Add tests or assertions proving the fixes still use the actual Core2 provider route and that the repaired cases stay green.

## Verification

- `.venv/bin/python -m pytest tests/agent -k 'core2 and (memory_manager_e2e or proof_harness or benchmark)' -n 0`
- Run a tiny benchmark smoke sample and confirm:
  - Core2 retrieval still appears in the real path
  - answer latency is lower or at least attributable
  - no fake model-only shortcut was introduced
