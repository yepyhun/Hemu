# Phase 04.9: MemPalace Codemap And Adoption Matrix - Research

## User Constraints

### Locked Constraints

- The user does not want “copy as much as possible” as the planning principle.
- The real question is:
  - where is `MemPalace > Core2`
  - where is `Core2 > MemPalace`
  - where is `Core2 + MemPalace > either alone`
- The result must stay implementable. A hybrid that collapses under complexity is a failure.
- The phase must preserve optionality. Later phases after 04.9 are conditional, not mandatory.

## Current Code Reality

- Core2 now has a closed deterministic truth/state/answer boundary and a green local proof for the wider failed cluster.
- MemPalace presents a strong retrieval/storage architecture:
  - verbatim drawers
  - semantic search with wing/room filtering
  - L0-L3 memory stack
  - AAAK compression / wake-up bootstrap
  - local SQLite temporal KG
  - room-based navigation graph
- MemPalace benchmark material is primarily retrieval-oriented, especially around LongMemEval recall rather than full end-to-end answer correctness.

## Source-Based MemPalace Findings

- `miner.py`: files verbatim chunks into drawers, routes content into rooms, and does not rely on summaries as the primary stored substrate.
- `searcher.py`: semantic search returns verbatim drawer content and supports wing/room filters.
- `README.md` and `layers.py`: define an explicit L0/L1/L2/L3 stack with wake-up context and on-demand deeper search.
- `knowledge_graph.py`: provides temporal triples with invalidate/query behavior, but this appears as a local graph utility rather than a Core2-style deterministic truth kernel.
- `palace_graph.py`: navigation graph is built from room metadata and tunnel/hall relationships.
- `mcp_server.py`: MemPalace exposes read/write tools and teaches its own protocol/AAAK surface to the agent.
- `benchmarks/longmemeval_bench.py`: benchmark focus is retrieval quality against LongMemEval rather than the full Core2-style authoritative answer path.

## Recommended Implementation Shape

Do not plan an immediate technical merge.

Plan 04.9 should instead create the decision artifacts that later phases depend on:

1. a MemPalace codemap normalized into a Core2 comparison ontology
2. an adoption matrix with explicit `adopt / adapt / keep / undecided` verdicts
3. a staged branch decision for what the next implementation phase should be, if any

## Best 3-Plan Split

### Plan 04.9-01

Codemap MemPalace and normalize the comparison layers.

### Plan 04.9-02

Create the adoption matrix and single-owner layer map.

### Plan 04.9-03

Derive the follow-on branch decision and stop rules for later implementation phases.

## Testing Guidance

- This phase is proven by artifact quality and internal consistency, not by runtime benchmark numbers.
- Every claimed MemPalace advantage should have either:
  - a concrete code anchor
  - or a clearly labeled README-level claim
- Every matrix row should say whether the verdict is:
  - strong evidence
  - moderate evidence
  - weak evidence

## Risks And Failure Modes

- Blind replacement drift: treating MemPalace retrieval strength as proof of total kernel superiority.
- Hybrid sprawl: authorizing multiple overlapping “smart” layers without ownership boundaries.
- Marketing capture: planning from README claims alone without code anchors.
- False symmetry: assuming every MemPalace layer must have a Core2 counterpart or vice versa.
- Premature commitment: hard-coding future phases before the adoption matrix actually supports them.

## Open Questions

- Can MemPalace wake-up layering be borrowed without committing to AAAK as a first-class Core2 representation?
- Is MemPalace’s temporal KG best treated as:
  - secondary navigation substrate
  - optional evidence source
  - or something to reject outright for Core2 truth/state work?
- After 04.9, is the best next move:
  - small targeted borrowing
  - bounded hybrid substrate work
  - or a broader paid rerun first?
