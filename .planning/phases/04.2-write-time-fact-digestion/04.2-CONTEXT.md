# Phase 04.2: Write-Time Fact Digestion - Context

**Gathered:** 2026-04-06
**Status:** Ready for planning

<domain>

## Phase Boundary

Phase 04.2 exists because Phase 04.1 proved the dominant remaining problem is structural: too much truth-building still happens at query time over raw session blobs. This phase moves the highest-value truth-building work into write-time or background digestion so later recall no longer has to improvise basic personal/update facts from raw history.

This phase must deliver a narrow, production-safe digestion layer for common durable memory cases:

- explicit durable remembers
- turn-ingested personal/profile facts
- current vs previous/update facts
- simple relations and compact fact summaries

This phase is not a benchmark-tuning phase, not a broad retrieval rewrite, and not a resurrection of the old kernel's large semantic/objective/admin stack.

</domain>

<decisions>

## Implementation Decisions

### Locked Decisions

- This phase is correctness-first and architecture-first; it exists to remove query-time truth-building load, not to add more query-time heuristics.
- `plan7vegrehajt.md` remains the primary product/system contract. `plan6.md` remains the hard guardrail against overcomplicated, self-collapsing design.
- Borrow patterns from the old kernel only where the codemap shows a clear net win. Do not import whole old subsystems.
- The digestion layer must stay local-first, modular, and inspectable inside `agent/core2_*` and the Core2 provider boundary.
- Prefer extending the existing Core2 planes, state model, and metadata over inventing a second parallel persistence architecture.
- V1 digestion should stay intentionally narrow: typed attribute/profile facts, update/currentness facts, simple relation facts, and compact delivery-ready summaries.
- History must remain inspectable. Write-time digestion may promote, supersede, or connect facts, but it must not silently erase earlier truth.

### the agent's Discretion

- Exact helper-module split, for example whether digestion lives in `core2_runtime.py` or a new `agent/core2_digestion.py`
- Exact metadata shape for fact kind, subject, predicate/relation key, normalized value, and digest provenance
- Whether compact delivery-ready artifacts are persisted in `delivery_views`, `derived_propositions`, or canonical metadata, as long as the result reduces later query-time work without architecture sprawl

</decisions>

<specifics>

## Specific Ideas

- Introduce typed digested fact records with stable metadata such as `fact_kind`, fact key, normalized value, currentness/update markers, and provenance links.
- Keep `raw_archive` as provenance anchor, but create smaller canonical fact objects that later phases can recall directly.
- Resolve common personal/update truth at write time so query-time recall no longer has to infer it from a whole conversation blob.
- Build compact delivery-ready summaries or retrieval hints during digestion when that reduces later query-time work without creating a second giant subsystem.

</specifics>
