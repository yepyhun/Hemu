# Phase 04.2: Write-Time Fact Digestion - Research

**Date:** 2026-04-06
**Status:** Ready for planning

<user_constraints>

## User Constraints

### Locked Constraints

- The target remains a production-grade, SOTA-competitive memory kernel, not a note bucket or benchmark-only patch.
- `plan7vegrehajt.md` is still the primary execution spec; `plan6.md` remains a hard guardrail.
- The user explicitly rejected endless query-time heuristic loops; if a structural gap is visible, planning must move the work to the deeper layer instead of adding more surface tricks.
- Core2 must stay modular and upstream-friendly; avoid recreating the old kernel's collapse-by-complexity pattern.
- Accuracy, provenance, and durable truth discipline outrank benchmark cosmetics in this phase.
- This phase should prepare speed indirectly by shifting work earlier, not by clipping or benchmark-specific hacks.

</user_constraints>

<phase_scope>

## Phase 04.2 Must Deliver

- A bounded write-time digestion path for the most valuable durable memory cases
- Typed fact records that are smaller and more queryable than raw session blobs
- Write-time handling of current vs previous/update semantics for those fact records
- Provenance-preserving links between raw archive, digested facts, and superseded/current records
- Compact recall-ready artifacts that later phases can use for faster fact-first retrieval

</phase_scope>

<current_state>

## Current Code Reality

Core2 already has most of the substrate needed for digestion:

- `agent/core2_store.py` already persists `raw_archive`, `canonical_truth`, `derived_propositions`, `retrieval_indices`, and `delivery_views`.
- `agent/core2_store.py` already knows about canonical lifecycle, supersession, conflict edges, and temporal metadata.
- `agent/core2_runtime.py` already has the write-path hooks: `remember(...)` and `ingest_turn(...)`.
- `plugins/memory/core2/__init__.py` already wires `core2_remember` and `sync_turn(...)` into the Hermes provider lifecycle.
- `agent/core2_authoritative.py` and current recall behavior already showed that fast answers are possible when smaller grounded structures exist.

What is still missing is the structural middle layer:

- `remember(...)` stores durable content but does not yet decompose it into typed fact objects for later fast recall.
- `ingest_turn(...)` persists turn history but does not yet produce durable fact records for common personal/update cases.
- Query-time still has to recover too much truth from large content blobs, which is exactly the bottleneck Phase 04.1 exposed.

</current_state>

<legacy_guidance>

## Non-Negotiable Lessons From The Legacy Kernel

The codemap says the old kernel still had a few net-positive patterns:

- compact answer packet
- direct resolved-answer fast path
- bounded compare/timeline routing
- explicit abstention/support discipline

But the old kernel collapsed under large additional layers:

- semantic-contract sprawl
- objective execution stacks
- admin/nightly/corpus pipelines that were too wide for the actual delivery value
- large graph subsystems that exceeded their net gain

Therefore Phase 04.2 should borrow only the narrow idea of doing more work before query time. It should not import the old subsystem shape.

</legacy_guidance>

<recommended_structure>

## Recommended Implementation Shape

Make the smallest strong architectural move:

1. Add a bounded digestion helper that can turn a durable memory input or a finished turn into typed fact candidates.
2. Persist those candidates using the existing Core2 planes and state model rather than inventing a second persistence family.
3. Resolve current vs previous/update semantics during digestion for the covered fact types.
4. Emit compact recall-ready artifacts or metadata so later recall can prefer the digested structures over raw blobs.

The intended narrow Phase 04.2 fact families are:

- profile/attribute facts
- explicit current/previous/update facts
- simple relation facts
- compact fact summaries for later direct-answer use

This phase should not yet promise a complete retrieval reroute across every query family. That belongs to the next structural phase.

</recommended_structure>

<plan_breakdown>

## Best 3-Plan Split

### Plan 04.2-01
Implement a bounded write-time digestion pipeline that turns `remember(...)` and `ingest_turn(...)` inputs into typed fact candidates with provenance.

### Plan 04.2-02
Implement write-time canonicalization for those fact candidates: current vs previous/update resolution, supersession, relation linking, and compact fact identity.

### Plan 04.2-03
Persist delivery-ready fact views or retrieval-ready metadata, then add proof/regression tests showing the digested layer exists, stays inspectable, and is ready for the later fact-first recall phase.

This order follows the structural fix:
1. create typed digested facts
2. make them durable and current-aware
3. make them ready to consume and prove they are real

</plan_breakdown>

<risks>

## Risks And Failure Modes

- Overreaching into universal extraction would recreate the legacy complexity trap.
- If fact metadata is too vague, later recall will still fall back to blob search and the phase will not pay off.
- If write-time canonicalization silently overwrites history, this phase would violate Core2 temporal guarantees.
- If delivery-ready artifacts become a second truth source instead of a projection of canonical truth, the design will split into competing layers.

</risks>

<open_questions>

## Open Questions

- Which exact fact families should be in the first digestion pass so the phase stays strong but narrow?
- Should compact fact views live in `delivery_views`, `derived_propositions`, or canonical metadata for the first implementation?
- How much relation-building is a net win in this phase before it starts resembling the old graph subsystem?

These are implementation choices, not blockers. The phase should make the smallest durable move that removes raw-blob dependence for common durable-memory cases.

</open_questions>
