# Plan 04.2-03: Materialize Recall-Ready Fact Views And Prove The Layer

## Objective

Create compact recall-ready fact artifacts from the digested canonical layer and prove they exist, remain inspectable, and are ready for the later fact-first recall phase.

## Must Haves

- Compact fact views or retrieval-ready metadata built at write time
- A clear handoff to the next phase, where recall will start preferring this layer
- Active tests proving the digested layer exists and is useful
- No benchmark-only shortcuts and no reintroduction of raw-blob dependence for the covered cases

## Tasks

- Decide the narrowest storage target for recall-ready artifacts:
  - `delivery_views`
  - `derived_propositions`
  - canonical metadata
- Materialize compact fact views that later phases can consume directly for:
  - profile/attribute lookup
  - current/previous/update questions
  - simple relation questions
- Extend explain/proof surfaces so a developer can inspect the digested record, its canonical state, and the compact recall-ready artifact.
- Add focused regression coverage showing the new artifacts are produced from write-time digestion and do not depend on rediscovering the same fact from raw session blobs.
- Update any local proof assets needed so the next phase can build fact-first recall on a real substrate rather than another planned abstraction.

## Verification

- Run targeted tests proving compact recall-ready artifacts are materialized during write time for covered fact types.
- Confirm those artifacts remain grounded in canonical truth and provenance instead of becoming a competing truth source.
- Confirm the phase leaves a clean handoff to the next GSD phase: fact-first recall.
