# Phase 08.1: Invariant Harness And Noise Repair Import - Research

**Date:** 2026-04-07
**Scope:** planning guidance only

## What The Legacy Agent Offers

The legacy Hermes agent has three relevant ideas near this scope:

1. architect-level memory invariants bound to explicit acceptance scenarios
2. a narrow acceptance harness that checks those invariants against fixture-backed flows
3. a repair pass that obsoletes obvious tool/file error artifacts mistakenly stored as active memory

The first two reduce ambiguity in future Core2 changes. The third improves long-term memory hygiene without changing the core truth model.

## Planning Implications

### What This Phase Should Build

Build one Core2-native hardening layer that:
- encodes a small number of high-value invariants
- checks them through repeatable local acceptance coverage
- can repair clearly bad structured-memory artifacts

### What This Phase Must Not Build

- a full legacy decision-memory runtime import
- a second truth engine
- fuzzy cleanup over subjective “low quality” memories
- retrieval ranking or routing retuning in the same phase

## Suggested Imported Invariants

The likely Core2-adapted invariants are:

- structured canonical facts must remain primary truth over fallback/native-style memory echoes
- the system must not claim a memory save or update without a real backing write path
- mutually contradictory active structured states must not remain simultaneously canonical when a deterministic resolution path exists

The phase should keep the list short. Fewer hard rules that are actually enforced are better than a large checklist that drifts.

## Suggested Noise Repair Scope

The repair path should only target obviously bad artifacts such as:
- tool error payloads accidentally stored as active memory
- file error payloads accidentally stored as active memory

The phase should not try to infer broader semantic quality or subjective usefulness.

## Cheap, Honest Proof

Recommended proof shape:
- fixture-backed invariant scenarios
- fixture-backed noise records
- one local outcome file with:
  - invariants checked
  - scenarios passed/failed
  - records repaired/obsoleted
  - verdict: `go`, `hold`, or `stop`

## Risks

### Biggest Failure Mode

The phase expands into a general memory-governance rewrite instead of staying a bounded hardening import.

### Secondary Failure Mode

The repair path grows fuzzy and starts obsoleting records that are merely awkward instead of clearly invalid.

## Guardrails

- Keep the invariant set very small.
- Keep the repair detector explicit and narrow.
- Keep all imported logic Core2-native at the seam, not as a wholesale legacy runtime transplant.
