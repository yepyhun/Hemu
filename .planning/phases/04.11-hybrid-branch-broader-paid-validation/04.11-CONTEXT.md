# Phase 04.11: Hybrid Branch Broader Paid Validation - Context

## Why This Phase Exists

Phase 04.10 proved the bounded hybrid substrate locally, but that proof is not enough to claim end-to-end improvement. The next honest step is to re-measure the hybrid branch on a broader paid validation run and decide whether the branch deserves another bounded follow-on phase.

## Inputs

- Phase 04.10 is execute-complete and UAT-complete.
- The current Core2 sampled baseline already closed a paid `10/10` run before hybrid work.
- A broader pre-hybrid paid `20` run previously exposed that local proof and sampled success do not automatically generalize.
- The 04.6/04.7/04.8 anti-loop rules remain active:
  - no benchmark-smink
  - no family spam
  - no soft comparator drift
  - no mixing retrieval-only wins with end-to-end correctness claims

## Locked Decisions

- This phase is validation-and-decision work, not a hidden implementation phase.
- The bounded hybrid seam from 04.10 must remain unchanged during the planned validation run unless a hard blocker prevents the run itself.
- Retrieval-only gains do not count as proof of full memory improvement.
- The paid protocol must stay comparable to earlier evidence, especially the prior `20`-question confirmation run.
- The paid run must be locked by a frozen run manifest before execution starts.
- The phase must end from one canonical machine-readable outcome file rather than scattered status truth.
- The phase must end in a bounded verdict:
  - go
  - no-go
  - mixed / stop and classify

## What This Phase Must Not Become

- no new substrate borrowing
- no new answer-surface widening
- no new family additions
- no comparator expansion
- no post-hoc threshold moving after the paid run
- no post-hoc run-condition drift
