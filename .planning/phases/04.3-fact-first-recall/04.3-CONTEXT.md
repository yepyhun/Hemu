# Phase 04.3: Fact-First Recall - Context

**Gathered:** 2026-04-06
**Status:** Ready for planning

<domain>

## Phase Boundary

Phase 04.3 exists because Phase 04.2 created a real write-time fact substrate, but the live recall path still enters through broad canonical search first. This phase makes covered recall paths prefer the new fact substrate directly, with explicit raw fallback only when fact evidence is insufficient.

This phase must deliver fact-first recall for the narrow, already-supported durable-memory cases:

- profile and attribute facts
- current vs previous/update facts
- simple relation facts
- compact fact views already materialized in Phase 04.2

This phase is not a new digestion phase, not a universal semantic retrieval rewrite, and not the final benchmark gate.

</domain>

<decisions>

## Implementation Decisions

### Locked Decisions

- Covered recall paths must prefer the Phase 04.2 digested fact layer before generic raw-blob-style canonical retrieval.
- Raw/canonical fallback remains mandatory when fact-first retrieval does not have enough support; fact-first is a preference, not a blind hard cut.
- The fallback path must preserve existing provenance, abstention, and current/previous semantics rather than weakening them for speed.
- Fact-first behavior must stay inspectable in code and proof artifacts. It cannot be a hidden heuristic inside ranking noise.
- Reuse the existing Core2 store/runtime/provider structure. Do not build a second retrieval architecture beside it.
- Borrow only narrow legacy patterns already validated by the codemap: compact answer packet, direct resolved-answer fast path, bounded compare/timeline routing.
- Keep scope intentionally narrow to the fact families already materialized in Phase 04.2.

### the agent's Discretion

- Exact store/runtime split for fact-first retrieval helpers
- Whether fact-first candidate retrieval lives as a new store API, a runtime helper, or both
- Exact route-family coverage inside Phase 04.3, as long as covered cases clearly become fact-first and unsupported cases still fall back safely

</decisions>

<specifics>

## Specific Ideas

- Add explicit fact-first retrieval helpers keyed by `fact_key`, `fact_compact` views, and digested metadata.
- Make runtime recall try the fact substrate first for covered query families, then fall back to generic canonical retrieval only if needed.
- Keep the authoritative answer path aligned with the same fact-first packets so covered fast answers become simpler and cheaper.
- Make proof surfaces able to show when fact-first retrieval fired and what fact artifacts were used.

</specifics>

<deferred>

## Deferred Ideas

- Universal fact-first routing for every query family
- Broad graph-first exploration redesign
- Final paid LongMemEval rerun
- Any LLM-heavy extraction or retrieval layer beyond the already materialized narrow fact substrate

</deferred>
