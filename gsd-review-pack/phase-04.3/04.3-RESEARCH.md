# Phase 04.3: Fact-First Recall - Research

**Date:** 2026-04-06
**Status:** Ready for planning

<user_constraints>

## User Constraints

### Locked Constraints

- This phase must keep following the GSD path cleanly and expose the next step explicitly.
- The user does not want another heuristic query-time loop. The move must be structural and inspectable.
- The old kernel can only be mined for narrow patterns, never re-imported as a large subsystem.
- Correctness and trust discipline remain above benchmark cosmetics.
- The phase should finish the logical next move after 04.2, not wander into unrelated optimization work.

</user_constraints>

<phase_scope>

## Phase 04.3 Must Deliver

- Fact-first candidate retrieval for the covered durable-memory fact families
- Safe fallback from fact-first retrieval to broader canonical search when facts are insufficient
- Runtime recall behavior that visibly uses the Phase 04.2 substrate
- Proof/regression coverage showing fact-first recall is real and does not break earlier recall contracts

</phase_scope>

<current_state>

## Current Code Reality

Phase 04.2 already delivered the substrate:

- `agent/core2_digestion.py` materializes narrow typed facts
- `agent/core2_runtime.py` writes those facts during `remember(...)` and `ingest_turn(...)`
- `agent/core2_store.py` now emits:
  - `fact_key` retrieval indices
  - `fact_compact` delivery views
  - same-identity supersession for changed current facts
- `core2_explain` can already show delivery views and retrieval indices

What is still missing is the recall handoff:

- `core2_runtime.recall(...)` still begins from generic `search_canonical(...)`
- `search_canonical(...)` still ranks mostly from broad lexical evidence over canonical content
- fact records help only if they happen to win that broad search
- authoritative answers still consume the same broad packet path first

</current_state>

<legacy_guidance>

## Non-Negotiable Lessons From The Legacy Kernel

The old kernel showed a real net win when:

- compact resolved answers could short-circuit later work
- compare/timeline questions used bounded, disciplined retrieval

It failed when:

- new retrieval layers multiplied without a clear substrate
- large semantic/objective stacks became the real product instead of helping the product

So Phase 04.3 should do one narrow thing well:

- route covered recall through the existing fact substrate first

It should not attempt a grand retrieval renaissance.

</legacy_guidance>

<recommended_structure>

## Recommended Implementation Shape

Make the smallest strong retrieval move:

1. Add explicit fact-first retrieval APIs over the existing 04.2 artifacts.
2. Teach runtime recall which query families should try those APIs first.
3. Keep raw canonical fallback when fact-first evidence is absent or incomplete.
4. Align the authoritative/direct-answer path with the same fact-first packet shape.

The strongest first coverage is:

- personal/profile attribute questions
- current vs previous/update questions
- simple relation questions already expressible from the digested layer

This phase should not yet promise perfect performance on every recall mode. It should make covered queries structurally cheaper and cleaner.

</recommended_structure>

<plan_breakdown>

## Best 3-Plan Split

### Plan 04.3-01
Add store/runtime retrieval primitives for fact-first lookup using `fact_key` indices, digested metadata, and `fact_compact` delivery views.

### Plan 04.3-02
Integrate fact-first routing into runtime recall and authoritative answering, with explicit fallback to broader canonical retrieval when fact evidence is insufficient.

### Plan 04.3-03
Add regression/proof coverage that demonstrates fact-first recall on covered queries, preserves previous contracts, and improves local payload discipline for those cases.

This order follows the architecture:
1. create the retrieval primitives
2. wire them into runtime behavior
3. prove they are active and safe

</plan_breakdown>

<risks>

## Risks And Failure Modes

- If fact-first retrieval becomes too aggressive, it could hide evidence that still only exists in broader canonical content.
- If the fallback path is unclear, the system may regress into either over-abstention or hidden heuristics.
- If the phase touches too many query families at once, it risks turning into another sprawling retrieval rewrite.
- If proof only checks correctness but not path activation, the phase may appear complete without actually using the new substrate.

</risks>

<open_questions>

## Open Questions

- Which query families in Phase 04.3 should be covered first so the win is real but the scope stays narrow?
- Should fact-first retrieval return canonical fact objects directly, or should it prefer `fact_compact` views and then rehydrate only when needed?
- How much of the authoritative path should move in Phase 04.3 versus remain as-is until a later fast-answer phase?

These are implementation choices, not blockers. The phase should make the smallest durable move that proves the 04.2 substrate is now actually being used.

</open_questions>
