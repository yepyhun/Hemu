# Plan 04.3-01: Add Fact-First Retrieval Primitives

## Objective

Create explicit store/runtime retrieval primitives that can look up covered facts through the new Phase 04.2 substrate before generic canonical search is attempted.

## Must Haves

- Fact-first lookup over `fact_key` indices and digested metadata
- Compact fact-view access for covered fact records
- Clear inspectability of which facts were selected
- No separate parallel retrieval stack

## Tasks

- Inspect the existing 04.2 artifacts in `agent/core2_store.py` and identify the narrowest APIs needed for fact-first retrieval.
- Add store helpers for covered fact lookup, for example:
  - by `fact_key`
  - by attribute/value aliases
  - by current/previous temporal slot
  - by relation key where already supported
- Add runtime helpers that can build a covered fact candidate set before the broader canonical search path runs.
- Keep results in the same Core2 record shape so later runtime code does not fork into a second contract.
- Extend explain/proof visibility if needed so it is obvious when a candidate came from the fact substrate.

## Verification

- Run targeted tests proving covered fact lookups can be served from the new substrate directly.
- Confirm no new sprawling retrieval architecture was introduced.
- Confirm fact-first candidate selection is inspectable in code and proof surfaces.
