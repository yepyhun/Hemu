# Phase 04.4: Uncovered Durable Memory Families - Context

**Gathered:** 2026-04-06
**Status:** Ready for planning

<domain>

## Phase Boundary

Phase 04.4 exists because Phases 04.2 and 04.3 fixed the architecture for a narrow set of durable-memory facts, but the paid live gate still exposed a bounded uncovered set:

- preference, habit, and routine memory
- aggregate and count-style memory
- broader temporal-summary and elapsed-time memory

This phase must turn those families into first-class Core2 substrate coverage so later recall and answering no longer fall back to slow or generic query-time reasoning for them.

This phase is not:

- a universal semantic rewrite
- a full graph platform
- a multilingual overhaul
- the final paid LongMemEval gate itself

</domain>

<decisions>

## Implementation Decisions

### Locked Decisions

- This phase is structural, not benchmark-cosmetic. The target is to close the real substrate gaps exposed by the paid gate.
- `plan7vegrehajt.md` remains the primary product contract. `plan6.md` remains the anti-loop and anti-collapse guardrail.
- The new families must become explicit covered fact families inside the existing Core2 architecture: write-time digestion, canonical state, retrieval indices, delivery views, and fact-first recall.
- Query-time regex or raw-blob parsing must not become the primary solution for these families.
- Reuse the existing `agent/core2_*` modules and provider seam. Do not build a second memory system beside Core2.
- Preference/habit/routine coverage must stay grounded and inspectable; it cannot degrade into open-ended “advice generation.”
- Aggregate/count coverage must be provenance-preserving and bounded; counts and totals should come from structured artifacts, not broad rescans of raw history.
- Temporal-summary coverage must stay bounded to common durable-memory cases such as elapsed time, before/after anchors, and compact event summaries. Do not recreate the old kernel’s giant reasoning stack.
- For the new families, direct fast answers must only fire from structured fact-first packets or equivalent structured artifacts. They must not silently fall back to raw-language shortcuts and still look like first-class support.

### the agent's Discretion

- Exact typed shapes and metadata for the new fact families
- Whether aggregate/count answers are backed by compact summary records, explicit membership facts, or both
- Whether temporal-summary support is best represented as event-anchor facts, compact timeline summaries, or a narrow combination

</decisions>

<specifics>

## Specific Ideas

- Add preference/habit/routine fact coverage for cases like evening preferences, sleep-related constraints, and stable do/don’t patterns.
- Add aggregate/count fact coverage for cases like distinct service counts, bounded totals, and compact multi-item summaries.
- Add temporal-summary/event-anchor coverage for cases like elapsed days between two remembered events.
- Add one small coverage contract so each newly covered family is provably wired end-to-end through digestion, canonicalization, fact-first recall, authoritative answering, and proof.
- Persist a tiny machine-readable gate status artifact so later sessions do not need to rediscover the last smoke/paid-gate state by hand.
- Keep proof tightly connected to the paid-gate failures that exposed these gaps.

</specifics>
