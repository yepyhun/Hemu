# Phase 04.4: Uncovered Durable Memory Families - Research

**Date:** 2026-04-06
**Status:** Ready for planning

<user_constraints>

## User Constraints

### Locked Constraints

- The user wants the kernel itself to become actually usable in real life, not another benchmark-only patch loop.
- The phase must stay on the GSD path cleanly and keep the next step explicit.
- The user explicitly rejected endless query-time heuristic growth; if a family is still missing, the fix must go to the deeper Core2 layer.
- The old kernel can only be mined for narrow net-positive patterns. No semantic-contract or admin-platform revival.
- The result should simplify later maintenance, not create another collapse-by-complexity path.

</user_constraints>

<phase_scope>

## Phase 04.4 Must Deliver

- Structural coverage for preference, habit, and routine memory as typed durable-memory facts
- Structural coverage for aggregate and count-style memory as typed durable-memory facts or summaries
- Structural coverage for bounded temporal-summary and elapsed-time memory
- Fact-first recall and direct-answer compatibility for the new families
- Proof/regression anchors tied to the paid-gate failure families so 04.1 can return to a meaningful final gate rerun

</phase_scope>

<current_state>

## Current Code Reality

Phases 04.2 and 04.3 already delivered:

- bounded write-time fact digestion
- current/previous canonicalization for covered facts
- `fact_key` indices and `fact_compact` artifacts
- fact-first recall for a narrow set of durable-memory families

Current narrow covered families in `agent/core2_fact_registry.py` are still mostly:

- occupation
- residence
- timezone
- manager
- team

The paid gate exposed three remaining families that are still too dependent on broad query-time reasoning:

1. Preference/habit/routine
- Example family: evening activity preferences with sleep-related constraints and negative preferences such as avoiding phone or TV late in the day.

2. Aggregate/count
- Example family: “How many different types of food delivery services have I used recently?”

3. Broader temporal summary
- Example family: elapsed time between finishing a book and later attending an event tied to another remembered anchor.

The current aggregate shortcut in `agent/core2_authoritative.py` helped a specific live case, but it is intentionally narrow and should not be mistaken for the general substrate solution for 04.4.

</current_state>

<legacy_guidance>

## Non-Negotiable Lessons From The Legacy Kernel

The legacy codemap shows a clear split:

- keep:
  - compact answer packets
  - direct resolved-answer fast paths when they read from already-structured memory
  - bounded compare/timeline routing
- avoid:
  - semantic-contract explosion
  - giant objective-execution layers
  - graph/quality/admin platforms inside the kernel

So 04.4 should add only the smallest additional structure needed to make these uncovered families first-class citizens of the existing Core2 substrate.

</legacy_guidance>

<recommended_structure>

## Recommended Implementation Shape

Make one bounded structural extension:

1. Extend the covered fact registry with the new durable-memory families.
2. Materialize them during write-time digestion and canonicalization instead of waiting for recall-time rediscovery.
3. Extend fact-first recall and authoritative answering to consume those structured artifacts.
4. Prove the new families with focused tests and gate-adjacent examples, then hand control back to 04.1 for the final external rerun.

Recommended family shapes:

- preference/habit/routine:
  - positive preference
  - negative preference
  - temporal qualifier such as “evening”, “before 9:30 pm”, “recently”
  - reason or constraint when explicitly present

- aggregate/count:
  - explicit membership set or compact summary record
  - dedupe-safe evidence links
  - bounded “recent window” or scoped qualifier when present

- temporal summary:
  - compact event facts with stable anchors
  - enough normalized temporal metadata to answer common elapsed/before/after questions without rebuilding a full timeline from raw blobs

This phase should not promise universal reasoning. It should make the current gate failures structurally covered.

</recommended_structure>

<risks>

## Risks And Failure Modes

- If preference facts are too loose, Core2 may become a generic suggestion engine instead of a memory system.
- If aggregate/count facts do not preserve provenance and dedupe correctly, answers may look fast but drift from truth.
- If temporal support grows into a general reasoning layer, complexity will rise too fast and repeat the legacy collapse pattern.
- If 04.4 claims universal coverage, Phase 04.1 will become another loop instead of a bounded final gate return.

</risks>

<open_questions>

## Open Questions

- Should aggregate/count coverage prefer explicit member facts, compact summary records, or a hybrid with one canonical answer surface?
- How should preference conflicts be represented when a newer routine or negative preference partially overrides an older one?
- What is the narrowest temporal-summary representation that solves the gate failures without becoming a generic timeline engine?

These are implementation choices, not blockers. The phase should make the smallest durable move that covers the known missing families.

</open_questions>
