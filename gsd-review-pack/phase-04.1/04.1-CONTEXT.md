# Phase 04.1: Kernel Correctness And Live Gate Recovery - Context

## Phase Boundary

Phase 04.1 exists because the first real paid LongMemEval live run exposed that the current full Hermes path is not trustworthy enough as an external acceptance gate. The user correctly called out that the deeper problem may be kernel correctness itself, not just benchmark speed.

This phase is not a new architecture phase. It is an urgent decimal insertion after Phase 4 to restore confidence in kernel correctness first, then in the real external benchmark gate, before any stronger release or SOTA claim.

The phase must stay tightly scoped to:
- live failure diagnosis on the real Hermes path
- kernel correctness fixes and regression coverage for the discovered failure modes
- benchmark telemetry and observability
- paid LongMemEval rerun readiness on the real Hermes-integrated Core2 path

It must not sprawl into:
- redesigning Core2 memory semantics
- replacing the Hermes runtime architecture
- switching to a model-only benchmark shortcut

## Implementation Decisions

### Locked Decisions

- The benchmark target remains the full Hermes runtime path with `memory.provider=core2`, not a raw model-only path.
- Correctness comes before optimization; if the kernel is functionally wrong, that must be fixed before treating latency as the primary problem.
- The paid LongMemEval gate remains the final external acceptance gate.
- The observed problem must be decomposed into correctness, retrieval, answer-generation, and benchmark-path components rather than assumed to be “just speed.”
- The benchmark runner must separate kernel-local cost from answer-generation cost and judge cost so failures are diagnosable.
- The phase should introduce an explicit smoke gate before spending paid budget on a larger live sample.

### The Agent's Discretion

- Exact telemetry field names and JSON report layout
- Whether answer-generation fast-path changes live in the benchmark harness, config patching, or thin runtime knobs
- Whether the first rerun target is 3, 5, or another small smoke sample before the full 20-item gate

## Specific Ideas

- Add per-item timing splits for seed, retrieval, answer generation, judge, and total wall time.
- Turn the live-run observations into explicit failure cases that can be replayed locally through Hermes.
- Add correctness-first regression tests for the failure patterns before another paid larger sample.
- Emit an explicit failure taxonomy that distinguishes prompt miss, abstention, retrieval miss, answer-generation miss, and judge disagreement.
- Add a benchmark fast-path that keeps the Hermes route real but removes avoidable overhead before the live gate.
- Add an early abort/smoke gate so obviously bad runs stop after a few items instead of burning time and money.

## Deferred Ideas

- Cross-model benchmark comparisons beyond the current benchmark provider
- Broader dashboarding or persistent benchmark history store
- Any retrieval-engine redesign not directly required by the live gate
