# Phase 04.1: Kernel Correctness And Live Gate Recovery - Research

## User Constraints

### Locked Constraints

- The benchmark must validate the current full Core2 kernel through Hermes, not a stripped direct API harness.
- If the kernel is functionally wrong, that correctness gap must be fixed before treating benchmark speed as the primary issue.
- Existing legacy benchmark logic should be reused where it helps, but not copied blindly.
- If the live run is clearly going badly, the system should stop early and report the failure pattern.
- User-facing checkpoints and decisions should stay plain and pragmatic rather than overly technical.

## Phase 04.1 Must Deliver

- A diagnosable live-run harness with replayable failure cases
- Kernel correctness fixes and targeted regressions for the discovered failure patterns
- A cheaper and faster smoke-gate path before a paid larger sample
- A concrete rerun path for the live LongMemEval gate on the actual Core2 kernel

## Current Code Reality

- `agent/core2_longmemeval_benchmark.py` already has the right overall shape:
  - stratified sampling
  - temporary isolated `HERMES_HOME`
  - real `run_agent.AIAgent` usage
  - prompt capture
  - estimated token-savings calculation
  - yes/no judge path
  - early-abort logic after the first five failed items
- `scripts/run_core2_longmemeval_subset.py` already exposes the benchmark entrypoint.
- The failed live run showed multiple queries reaching `core2_rec` quickly, then spending most of the time in `run_conversation` / streaming answer generation.
- This strongly suggests that retrieval is not the first obvious bottleneck, but it does not prove overall kernel correctness. The next wave must turn live failures into replayable correctness cases before spending more on bigger runs.

## Existing Anchors Worth Reusing

- Old Hermes LongMemEval adapter:
  - stratified sample logic
  - answer-check prompt families
  - timing split semantics
  - temporary home isolation
  - per-item report structure
- Current Core2 proof assets:
  - `core2_proof_harness.py`
  - targeted pytest coverage proving provider/runtime/store correctness
- Current synthetic benchmark math:
  - replay/token-savings estimate already shows strong local compression characteristics, so the live bottleneck likely sits elsewhere

## Recommended Implementation Shape

### Plan 04.1-01

Instrument the live path and convert the failed run into actionable correctness evidence so every item clearly reports:
- seed time
- retrieval/tool time
- answer-generation time
- judge time
- total time
- token counts
- failure category

Also add a small smoke-gate mode that makes the paid run fail fast when the correctness or latency profile is obviously outside the target envelope.

### Plan 04.1-02

Fix the kernel behaviors exposed by the live path and add regressions for them. Reduce avoidable benchmark-path latency only after the failure cases are correctly understood.

Likely candidates:
- incorrect or weak answer synthesis after correct retrieval
- insufficient grounding propagation from retrieval to final answer
- query handling mismatches visible only on the real Hermes route
- smaller benchmark answer mode
- explicit benchmark config knobs
- optional judge deferral
- lower-overhead conversation path where safe

### Plan 04.1-03

Run the live gate again in stages:
- tiny smoke gate
- expanded local or paid subset
- final release decision artifact

The outcome must be inspectable even if the gate still fails.

## Testing Guidance

- Add tests around telemetry splitting and failure classification so the benchmark runner is not a black box.
- Add tests for the replayed live failure cases and their correctness fixes.
- Add tests for smoke-gate early-stop behavior.
- Keep the existing local pytest Core2 suite green while the benchmark tooling evolves.

## Risks And Failure Modes

- The true bottleneck may be answer synthesis, grounding handoff, or the answer model/streaming path rather than the benchmark wrapper, so the phase must not over-assume “performance-only.”
- Paid benchmark cost can rise quickly if smoke-gate logic is weak.
- If telemetry is incomplete, another failed live run will still not explain enough.
- Over-optimizing the benchmark harness into a fake path would invalidate the gate.

## Open Questions

- Which minimal Hermes runtime knobs can be safely exposed for benchmark mode without compromising correctness?
- Should the first paid rerun skip judging during the smoke gate and judge only passed candidates afterward?
- What exact latency target should be enforced for smoke-gate aborts per item and per mini-run?

These questions are not blockers. The phase should make the smallest strong move that produces a trustworthy rerun path.
