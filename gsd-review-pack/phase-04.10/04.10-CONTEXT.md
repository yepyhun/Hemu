# Phase 04.10: Bounded Hybrid Substrate - Context

## Phase Boundary

Phase 04.10 implements the recommended `bounded hybrid substrate` branch from Phase 04.9.

This phase is limited to:

- MemPalace-inspired raw/verbatim substrate work
- scoped retrieval under Core2 governance
- adapter seams that let Core2 consume that substrate without surrendering truth/state ownership

This phase must not:

- replace Core2 truth/state/provenance logic
- replace Core2 answer surfaces or abstention behavior
- introduce a second truth engine
- roll out AAAK broadly
- adopt graph/KG as primary truth
- replace the benchmark/evaluation harness

## Implementation Decisions

### Locked Decisions

- 04.10 is a substrate phase, not a whole-kernel rewrite.
- The owner of truth/state/provenance remains Core2.
- The owner of deterministic answer surfaces and abstention remains Core2.
- MemPalace-inspired pieces may be introduced only through a bounded adapter seam under Core2 governance.
- The hybrid seam must expose a tiny stable substrate interface to the rest of Core2 rather than leaking MemPalace-specific internal concepts upward.
- If the substrate cannot be integrated without ambiguous ownership, the phase must stop and fall back to a smaller borrowing path.
- Prefer shadow or bounded cutover semantics over one-shot full replacement.
- A clean rollback path must exist before the hybrid substrate is treated as a default path.
- The phase must preserve the existing deterministic-core boundary from Phase 04.6 and the routing/answer contract work from Phases 04.3 to 04.8.

### At the Agent's Discretion

- Whether the adapter is expressed as:
  - a new internal substrate module
  - a retriever helper
  - or a provider-internal service
- Whether the first integration point should enter through raw storage, scoped retrieval, or both at once, as long as the phase remains bounded and provable.
- Whether to leave a fully disabled experimental path in place for comparison, if it does not create dual ownership.

## Specific Ideas

- The likely first seam is:
  - MemPalace-style verbatim/raw substrate
  - plus wing/room-like scoped retrieval
  - while Core2 continues to own digestion, truth-state, and answer surfaces
- The safest shape for 04.10 is:
  - adapter seam first
  - shadow or bounded cutover second
  - proof and ownership checks before anything more ambitious
- A paired old-vs-new retrieval trace for the same query should exist so the hybrid path stays debuggable and reviewable.

## Deferred Ideas

- AAAK as a first-class Core2 representation
- MemPalace temporal KG as truth authority
- Palace graph/navigation as mandatory runtime dependency
- benchmark harness replacement
- wholesale MemPalace migration
