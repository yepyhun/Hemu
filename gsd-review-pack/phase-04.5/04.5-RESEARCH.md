# Phase 04.5: Deterministic Answer Surface And Gate Closure - Research

## User Constraints

### Locked Constraints

- The user wants a truly working kernel, not infinite benchmark-loop patching.
- Correctness-first remains mandatory; external benchmark reruns are for proving the finished path, not for discovering basic architecture by accident.
- The work must continue through the GSD workflow with explicit next-step guidance.
- Low-value extra complexity is forbidden. The phase must simplify the final handoff, not add another opaque layer.

## Phase 04.5 Must Deliver

Phase 04.5 must close the gap between “Core2 knows enough” and “the live Hermes path actually answers reliably.” That means:

1. Covered durable-memory families need a provider-owned answer surface that is narrower and more deterministic than a generic recall blob.
2. The Hermes handoff must consume that surface in a way that keeps the answer path inspectable.
3. The remaining gate failures must become machine-classified blockers instead of ambiguous benchmark anecdotes.

This phase does not need to make the entire benchmark green by itself. It needs to make the remaining rerun honest, inspectable, and worth trusting.

## Current Code Reality

- Core2 now has write-time digestion and fact-first recall for covered durable-memory families.
- Core2 also has guarded structured direct answers for several covered families.
- The real live gate improved materially but still reports `prompt_miss` as the dominant failure family.
- The remaining external failure is therefore no longer “the kernel knows nothing”; it is “the final answer handoff is still too loose.”
- The latest gate status already tells us the narrow target:
  - sample completed: 9/10
  - pass rate: 4/9
  - dominant failure pattern: `prompt_miss`
  - secondary blocker: one latency-heavy fallback path

## External Pattern Guidance

The external references all point in the same narrow direction:

- `veela`: the provider should own layered memory surfaces and prefetch a narrow recall block instead of leaving the final path to rebuild meaning from raw history.
- `mem0`: the runtime contract should look more like `add/search/use compact memory context` than `search everything and hope the model formats it correctly`.
- `PLUR`: durable memory and episodic history should stay separate; the prompt should receive ranked, bounded memory, not undifferentiated memory mass.
- `lynx`: the finish line must be mechanically verifiable. Tests and gate artifacts should define “done,” not descriptive language.

These references support a narrower handoff contract. They do not justify building a broader semantic platform.

## Non-Negotiable Lessons From The Legacy Kernel

- Good ideas worth keeping:
  - answer-packet discipline
  - bounded compare/temporal routing
  - direct grounded answer short-circuit
- Failure modes to avoid:
  - semantic-contract explosion
  - planner-on-planner objective layers
  - giant admin runtime
  - graph/quality/benchmark sprawl inside the kernel

Phase 04.5 should therefore make the answer handoff more deterministic without turning Core2 into a second orchestration platform.

## Recommended Implementation Shape

Make one bounded final architectural move:

1. Define a deterministic answer-surface artifact for covered durable-memory families.
2. Make Core2/provider code emit that artifact directly from the structured substrate.
3. Teach the Hermes handoff path and benchmark/runtime proof tools to consume and classify this artifact explicitly.
4. Prove the new path locally, then return control to Phase 04.1 for the real staged live rerun.

Recommended answer-surface layers:

- `fact_only`
  - one deterministic answer line or compact structured field set
  - strongest provenance/support
- `fact_plus_summary`
  - same structured fact answer plus a short explanatory/support surface
- `fallback`
  - explicit signal that the provider could not produce a deterministic answer surface and broader answering is required

Operational guardrail:

- the answer surface should have an explicit small-size ceiling
- this ceiling is not a new token-budget subsystem; it is a simple anti-drift rule so the surface cannot quietly become another recall blob

Recommended gate matrix:

- `kernel_correctness`
  - retrieved/structured truth is wrong or missing
- `handoff_format`
  - structured truth exists but the final answer handoff misses or misformats it
- `latency`
  - path is correct enough but too slow because it falls back to broader answering
- `unknown`
  - evidence is not strong enough to classify the failure cleanly, so the classifier must fail closed instead of inventing confidence

## Best 3-Plan Split

### Plan 04.5-01

Define and emit deterministic answer-surface artifacts from Core2/provider code for covered durable-memory families.

### Plan 04.5-02

Wire the Hermes handoff and direct-answer path to prefer the new answer surface and to preserve explicit family/failure metadata.

### Plan 04.5-03

Add machine-readable gate classification, focused proofs, and hand back to 04.1 for the real staged live rerun.

## Testing Guidance

- Add focused tests that prove the answer surface exists and is emitted for covered families.
- Add or extend tests that show direct answers only come from structured surfaces, not from unproven prompt-like fallbacks.
- Extend gate-status tests so failures are classified into stable blocker families.
- Keep a tiny frozen real-case canary replay set from the existing live failures so 04.5 can prove handoff improvements without spending the paid gate every time.
- Keep a narrow smoke proof through the actual Hermes/Core2 path; do not claim the paid gate is complete inside 04.5 itself.

## Risks And Failure Modes

- The answer surface becomes too large and quietly turns into another prompt blob.
- The handoff becomes benchmark-specific instead of provider/runtime-specific.
- The classification artifact becomes fuzzy or subjective rather than machine-derivable.
- The classifier overstates confidence and hides ambiguity that should have remained explicit.
- This phase accidentally grows into a general reasoning layer instead of a final handoff contract.

## Open Questions

- What is the smallest answer-surface representation that still gives the final path enough structure to avoid `prompt_miss`?
- Which parts belong in provider/runtime output versus `core2_authoritative` versus benchmark/gate instrumentation?
- Should the direct answer surface render final text, structured answer fields, or both for the covered families?

These are implementation choices, not blockers. The phase should make the smallest deterministic move that closes the handoff gap.
