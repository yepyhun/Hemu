# Plan 04.5-01: Emit Deterministic Answer Surfaces

## Objective

Define and emit a narrow Core2 answer-surface artifact for covered durable-memory families so the real Hermes path no longer depends on a loose recall blob as its first-pass memory handoff.

## Must Haves

- The artifact is provider-owned, typed, and inspectable.
- It is derived from the existing fact-first substrate, not from a new raw-blob parsing path.
- It supports at least the covered durable-memory families already established by Phases 04.2-04.4.
- It remains small enough to be a real answer surface, not a hidden generic prompt context.

## Tasks

- Add or refine the Core2 types/answer modules so there is a stable answer-surface structure for covered durable-memory families.
- Include the minimum fields needed to make the handoff deterministic:
  - family
  - answer mode
  - direct answer text and/or compact structured answer fields
  - provenance/support metadata
  - explicit fallback signal
- Emit this surface from the Core2 runtime/provider path where covered facts are already available.
- Keep the surface aligned with fact-first metadata such as:
  - `fact_key`
  - retrieval path
  - support/provenance
- Do not let this step introduce new broad memory-family coverage or a second answer system beside Core2.

## Verification

- Focused tests prove that covered durable-memory queries now produce deterministic answer-surface artifacts.
- The emitted artifact is inspectable and stable enough to use in later handoff/gate classification.
- Existing retrieval and authoritative tests remain green.
