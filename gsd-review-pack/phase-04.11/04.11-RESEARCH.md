# Phase 04.11: Hybrid Branch Broader Paid Validation - Research

## User Constraints

- The user wants the GSD path kept explicit rather than letting a paid rerun float outside the roadmap.
- The user wants anti-loop discipline preserved.
- The user does not want retrieval-only improvements mislabeled as full memory-kernel wins.

## What 04.10 Actually Authorized

Phase 04.10 authorized a bounded hybrid substrate seam and local non-regression proof.

It did not authorize:

- claiming broad end-to-end success from local proof
- widening the seam during validation
- inserting another hybrid follow-on phase without stronger paid evidence

## Recommended Validation Shape

Use a stepwise but bounded validation protocol:

1. lock the validation protocol before running paid evaluation
2. run a cheap local prerequisite check
3. run the same broader paid validation shape that previously exposed the gap
4. classify the outcome against fixed rules

The protocol should include a tiny frozen run manifest so later comparisons cannot drift on unnoticed run conditions.

The most important comparison is against the earlier broader hybrid-predecessor evidence, not against README-level repo claims or retrieval-only scores.

## Best 3-Plan Split

### Plan 04.11-01

Define the paid validation protocol, comparison rules, and stop rules.

### Plan 04.11-02

Run local prerequisites plus the broader paid validation on the bounded hybrid branch.

### Plan 04.11-03

Classify the results into go / no-go / mixed and hand back a bounded next move.

## Testing Guidance

- Keep the hybrid runtime mode explicit during the run.
- Keep the paid sample comparable to prior broader evidence.
- Record both overall pass-rate and failure-class composition.
- Emit one canonical machine-readable outcome artifact and derive later summaries from it.
- Distinguish:
  - promptless authoritative wins
  - fallback/handoff wins
  - kernel-correctness regressions
  - mixed ambiguous outcomes
- Do not move thresholds after seeing the result.

## Risks And Failure Modes

- declaring victory from a retrieval-looking improvement that does not improve end-to-end outcomes
- rerunning with changed harness assumptions and breaking comparability
- quietly editing hybrid behavior mid-validation
- treating noisy mixed results as permission for more hybrid implementation
- spending more paid runs before a bounded verdict exists
- conflicting status files that make the run harder to interpret than it should be

## Open Questions

- Should the broader run stop at the comparable `20` set, or escalate only if the `20` result is cleanly improved?
- What exact pass/failure delta is strong enough to justify another bounded hybrid follow-on phase?
- If results are mixed, should the next move be rollback-to-shadow/off analysis or a narrower classifier phase?

## Verdict Priority

If signals conflict, classify in this order:

1. regression
2. mixed
3. go

Only clear non-regressing improvement earns `go`.
